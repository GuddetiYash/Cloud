Overview

The Run Configuration Panel allows you to Run PLEXOS Models using the most suitable configuration settings, such as Run Types, Engine Version, VM Type, and more. It provides organized cards for Study Recommendation, Study Default, and Previous Run Configuration, enabling you to quickly choose the desired configuration and run the model instantly.

 

The Run Configuration Panel also offers a customizable interface, where you can adjust the Model’s configuration settings, such as the Run Types, Engine Version, and VM Types, to match your specific needs. This flexibility ensures that models are run with the optimal settings for your convenience.

 

This help article describes each configuration setting and how to configure these settings to run the Model effectively.

Understanding the Run Configuration Panel

The Run Configuration Panel is located on the Models page in PLEXOS Cloud. However, you can also access the panel by clicking the Run action button in front of each model on the Overview, Simulations, Models, Solutions, or Simulation Analytics pages.

 

The layout of the Run Configuration Panel remains consistent across all PLEXOS Cloud interfaces. 

 

To open the panel on the Models page, perform the following steps:

 

  1. Sign In to PLEXOS Cloud using your credentials.

  2. Navigate to the Models page.

  3. Select the Model or multiple Models you wish to run.   After selecting the Model(s), the Run Models panel will appear in the top-right corner of the screen. 

  4. Drag the panel to the left to view the full Run Configuration Panel.


      NOTE: The Run Model panel is the same as the Run Configuration Panel.

 

When you select the Models to run, the Run Configuration Panel provides key information, such as available Credits, Concurrent Cores, the selected Model name, and Configuration cards like Previous Run Details, Previous Configuration, Recommendations, and Study Recommendations. Selecting one of these cards and clicking Run can trigger the Model Run instantly.

[B] Details of the Run Configuration Panel 

Run Configuration Panel:

 

Refer to the following points to understand the key points and information cards in detail.

 

  1. Credits: Displays the total available and consumed Credits in both number and percentage formats. This helps you manage Credit usage effectively.

  2. Concurrent Cores: Displays the total available and consumed Cores in both number and percentage formats, helping you manage Core usage and select the appropriate configuration based on your available Core limits.

  3. Model Name: Displays the name of the selected Model.

  4. Previous Run Details: Provides information on the last run, including the time it was executed, the Run Type, Engine Version, VM Type, Credits consumed, and access to Logs. This helps you review how the model was previously run and view logs instantly to gain insights.

  5.  Previous Configuration: This card displays the configuration settings used for the previous run of the selected model, including the Run Type, Engine Version, VM Type, and Solver. It also shows the required credits or cores if you choose to run the model again with the same configuration. This card allows you to quickly rerun the model using the same settings, saving time and effort.

  6. Recommendation Card: Shows system-generated recommendations based on the first run of the selected model. These recommendations aim to help you save cores and optimize resource usage. If there are no previous runs, the default recommendation is a Single Run Type and 8 Core – 64 GB VM Type.

  7. Study Recommendations: The Study Recommendation card displays the default configuration settings set for the selected Study, including the Engine Version, Virtual Machine Type, and Locale Settings. These defaults are configured on the Overview page and are only applicable to the Single Run Type. It allows you to quickly Run the Model using the pre-defined settings, ensuring consistency and saving time.

  8. Add to Simulation Group: This option allows you to add the selected Model to either the existing Simulation Group, or you can create a new Simulation Group and add it to that group.  A Simulation Group is a set of Simulations that runs together in a batch. 

    For more information, refer to the detailed Simulation Groups help article.  

  9. Customize Button: The Customize button is available on all Configuration Cards. If you need to adjust the settings, the Customize button allows you to modify the displayed configuration to meet your specific requirements.

[A] Run Type

The Run Type section allows you to select a preferred run type configuration that best suits your simulation needs. There are three primary run types available: Single Run Type, Distributed Run Type, and Tuning Run Type.

 

  1. Single Run Type: The Single Run Type allows you to Run selected Models on a single virtual machine. This option is ideal for simulations that don’t require distributed resources.

  2. Distributed Run Type: A Distributed Run enables you to control and maximize the computing resources for a Model Run. Currently, PLEXOS Cloud supports the below-mentioned Distribution Methods for Distribution Run type.

[C] VM Type

The VM Type section allows you to select the preferred virtual machine configuration for your Simulation. The required credits for a simulation run are determined based on the selected Engine Version and VM Type.

 

Choosing the appropriate engine version and VM type with sufficient memory, CPU cores, and disk space helps prevent common issues such as out-of-memory errors or insufficient core allocation.

 

PLEXOS Cloud supports two primary types of virtual machines:

 

  • Standard VM Type: Standard VMs have a regular Credit rate for running a Model. The Credit calculation for Simulation runs is based on a 5-minute interval block structure.

  • High Memory VM Type: The high-memory VMs have a higher Credit rate than standard VMs and are available upon request. In the Run Configuration panel, they are labeled as 1.5X, 2X, 3X, and so on, allowing you to distinguish between standard VMs and high-memory VMs.

    The high-memory VMs provide more RAM on specific core configurations, offering extra power for your Simulations. This allows you to Run long and memory-intensive Models smoothly without being limited by the memory constraints of standard VMs.  

    For example, if an 8-core Virtual Machine is labeled as 2X in the Run Configuration panel, it will consume twice the normal Credits for that Simulation. Generally, a standard 8-core Virtual Machine consumes 0.67 Credits for the first 5 minutes of a Simulation. If it is labeled as a high-memory VM with 2X, it will consume 1.34 Credits for the first 5 minutes of the Simulation.

 

  NOTE:  Please reach out to your Customer Success Manager to request high-memory Virtual Machines (VMs) if needed.

 

  TIP: The Run Summary panel on the right side of the screen provides real-time updates on the minimum required credits for the selected VM type. Refer to the Run Impact section within the Run Summary panel to view the required Credits for each VM type at a glance.

[D] Solution Chaining

The Solution Chaining section allows you to use Solutions from other Models as input for your current model. For example, if you want to use Solutions generated from Model A as input for your current Model B, you can utilize the Solution Chaining feature. Simply click Add Input to add Model A’s Solution files as input for Model B’s Run.

 

Solution Chaining supports adding multiple solutions from various Models, allowing you to configure comprehensive inputs for your current Model Run.

 

To Add Input in Solution Chaining, perform the following steps:

 

  1. Click ADD INPUT. 

  2. Select the desired Study, Model, Simulation (Date & Time), and the desired Solution from that Study.

[E] Additional Settings

The Additional Settings section allows you to configure various simulation settings and post-run actions, including Simulation Settings (Locale Settings), PLEXOS Launcher Parameters (Beta), Solution Reporting, Congestion Analysis Data, and preferences for downloading Solution Files.

[1] Simulation Settings

The Simulation Settings allow you to select Locale Settings and set the Max Process Inactivity. The Process Inactivity setting specifies the maximum time (in minutes) allowed for inactivity in the PLEXOS engine logs. The system will automatically cancel the Run if there is no activity during the set time.

 

[2] PLEXOS Launcher Parameters (Beta)

The PLEXOS Launcher Parameters field is an advanced Run parameter that allows you to insert command-line arguments to Run your Simulations. You can use these command lines to run PLEXOS Simulations, execute Models, and use Solution merging capabilities without accessing the PLEXOS user interface.

 

Common Scenarios for Using PLEXOS Launcher Parameters:

 

  • Control the number of instances to run simultaneously and determine the allocated cores.  
  • Perform and manage Split Runs. 
  • Decide on the output destination. 
  • Handle special configurations and solver parameters.
  • Set cultural preferences and more.

 

The PLEXOS Launcher Parameters feature requires the correct logical syntax in command lines for the successful execution of the Simulation. 

 

 

Key Highlights of using the PLEXOS Launcher Parameters:

 

  1. The PLEXOS Launcher Parameters field is only available for the Single Run type. 

  2. For all Simulations executed via PLEXOS Launcher Parameters:

    • Upon the successful Simulation Run, you can only download the generated artifacts from the Solutions page.  

    • When you Run the Simulations with PLEXOS Launcher Parameters, the system does not generate output in Parquet file format. As a result, PLEXOS Cloud Analytics capabilities, such as BI Analytics, Simulations Groups, Comparisons, View Map, Grid, etc., are unavailable.

    • The inserted command is viewable in the Custom Arguments column on the Simulations page.

 

  NOTE:  The PLEXOS Launcher Parameters feature requires advanced knowledge of PLEXOS Launcher command line syntax. Not all combinations of command-line arguments are supported, so it’s advisable to consult Energy Exemplar support before using this feature.

[4] Memory Management

The Memory Management section allows you to manually configure the Pagination Memory percentage for Virtual Machines (VMs). This setting is only available for the Single and Distributed Run Types using Linux Engines.

 

By adjusting this setting, you can avoid out-of-memory errors and run simulations more efficiently, particularly on lower-spec machines, without needing to switch to high-memory VMs. 

 

 

 NOTE: When configuring Pagination Memory for a Chronological Split Execution Run Type, the setting applies to both the parent and child run configurations.

 

➡️ What is Pagination Memory?

 

Pagination Memory refers to a portion of the system’s physical memory that is reserved to store in-memory data temporarily during simulation execution. Instead of running entirely in memory, part of the simulation data is paged to disk when needed, helping prevent crashes caused by limited RAM.

 

➡️ Example Use Cases


In many simulations, high memory usage is not constant; most of the run requires moderate memory, with only brief periods where memory demand temporarily spikes.


❇️ Scenario: For instance, if a simulation typically runs on a 16-core, 120 GB RAM VM but analysis shows that the average memory usage is around 64 GB, there is room to optimize.


 In such cases, 

 

  • Instead of continuing to run on a high-memory VM, you can switch to a lower-spec machine [e.g., 8-core, 64 GB RAM VM] and,
  • Increase the Pagination Memory percentage in the Run Configuration Panel.


❇️ Why This Works: 

 

  • A higher Pagination Memory allocation reserves more memory for active computation.
  • Excess data is offloaded to disk during spikes, preventing crashes.
  • Reduces dependency on high-memory VMs for short-term peaks.

 

➡️ Key Points to Remember

 

  • Applicable for both Single and Distributed Run Types on Linux Engines.
  • Can be configured from both the UI and CLI.
  • Pagination Memory value must be between 34% and 70%. Configurations outside this range, will result in an error.

 

[5] Re-Reporting Option

The Re-Reporting [with Solution Snapshot] option allows you to reuse an existing solution file when running a new simulation focused only on additional or modified reporting outputs.


These settings are available for Single Run Types and help reduce engine time, minimize cloud credit usage, and enable faster iteration during early-phase analysis.

 

The Re-reporing feature works with the  Solutions Snapshot parameter [available under the Solution Chaining section]. Refer to the following points for more information:

 

  • Re-Reporting: When enabled during a simulation run, it allows the simulation’s solution file to be referenced in a future run and report additional outputs faster.


  • Solution Snapshot Parameter: The Solution Snapshot option is available in the Select Solutions dropdown within the Solution Chaining section. This parameter is shown only for simulations run with Re-Reporting enabled and allows you to run a follow-up simulation that focuses only on newly added or missing reporting outputs without reprocessing the full dataset.

 

➤ How it works?

❇️ Use Case: Consider a scenario where Simulation A is run with a limited set of reporting properties, or some properties are accidentally missed. Later, you decide to run Simulation B to include additional reporting outputs, such as Generation, FO&M Cost, or any metrics not captured initially.

 

❇️ Solution: In this case, instead of re-running the entire simulation from scratch, you can follow these steps to reuse the existing solution file and generate only the new outputs:

 

  1. Enable Re-Reporting in Simulation A: While configuring Simulation A, turn on Re-Reporting in the Additional Settings section and complete the run.

  2. Update the Model with New Outputs: Add the missing or new reporting properties in PLEXOS Desktop and save the updated Model.

  3. Use Solution Chaining to Set Up Simulation B

    1. In the Solution Chaining section, select the correct artifacts, such as the same Study, Model, and Simulation A.

    2. From the Select Solutions dropdown, select the Solution Snapshot parameter.

  4. Run Simulation B.

 

❇️Result: The PLEXOS Engine reuses data from Simulation A and generates only the newly added reporting outputs, avoiding full reprocessing and saving compute resources.

[G] Run Summary

The Run Summary section provides a detailed overview of the impact of the selected configuration, including the initial Credits required based on the selected Engine Version and VM Type. This section also displays the Model name, License type (Credit or Core), and shows available vs. consumed Credits and concurrent Cores.

 

Details Available in Run Summary:

 

  • Run Impact: Displays the initial Credits required based on your configuration choices, helping you understand the resource consumption for your Simulation Run.

  • Model Name: Shows the name of the Model you have selected for the Run.

  • License Type: Indicates whether the simulation run uses a Credit-based or Core-based license.

  • Credits and Concurrent Cores: Provides information on the available Credits and Cores versus the amount already consumed, allowing for effective resource management.

 

Running a Model

Once you have customized the configuration, simply click the Run button to initiate the Model Run.

 

After clicking the Run button, the Model will be queued for execution. You can then monitor the status of your simulation from the Simulation History Table on the Simulation page. The Simulation page provides an overview of both in-progress and completed simulations, allowing you to track progress and review results efficiently.
